版本：V1.0（MVP）

作者：AI 助手（基于《项目规划.md》整理）

## 一、产品概述

### 1.1 产品定位

面向本地生活服务场景（如理发、洗车、美甲等），IntelligentLocalBid 是一个基于 **Multi-Agent 架构** 的智能出行顾问平台。  
用户通过自然语言输入出行或服务需求，系统由主 Agent（Orchestrator）进行任务规划，协调多个专职子 Agent 执行地点搜索、实时验证、评论聚合、摘要评分和个性化排序等任务，最终返回可解释的 Top N 个推荐结果，并在地图上进行可视化展示。

### 1.2 核心价值与差异化

- **对比通用大模型（Gemini / ChatGPT）**
  - 不只是输出“文字建议”，而是提供「地图 + 实时信息 + 评论聚合 + 个性化排序」的一体化垂直体验。
- **对比传统地图（Google Maps）**
  - 不只是按评分/距离排序，而是结合用户画像、预算、场景偏好做「个性化 Top N 排行榜」。
  - 提供 LLM 驱动的评论摘要与可解释的推荐理由。
- **技术亮点**
  - 使用 **LangGraph** 搭建有状态、多 Agent 的 DAG 调度工作流。
  - 展示多 Agent 并发执行、状态追踪和降级策略等工程能力。

### 1.3 文档目标

本 PRD 的目标是：

1. 对齐产品、研发、测试等角色对本项目的 **目标、范围和优先级**；
2. 明确 MVP 版本（一周冲刺）需要交付的 **核心体验与功能边界**；
3. 为架构设计和任务拆分提供自上而下的 **需求依据**。

## 二、目标与范围

### 2.1 业务目标（MVP，一周内）

1. 支持用户用中英文自然语言输入本地服务或出行需求，系统能正确理解意图并发起搜索。
2. 返回 **Top 10 个推荐地点**，每条包含名称、评分、距离、价格区间和简要推荐理由。
3. 在地图上可视化展示推荐地点，并与排行榜列表联动（点击/悬停联动高亮）。
4. 支持基础个性化能力：冷启动问卷 + 简化权重调节（价格/距离/评分等）。
5. 提供可在线访问的 PWA（扫码可用），用于演示完整端到端体验。

### 2.2 不在本期范围（V2 之后）

- 多数据源强一致性校验（Google Maps vs 商家官网/社交媒体 的差异对齐 Agent）。
- 复杂 AB 测试、精细化运营后台。
- 多城市大规模运营能力与复杂运营策略。
- 商家端管理后台（认领店铺、编辑详情等）。

## 三、用户与使用场景

### 3.1 目标用户

- 一线 / 新一线城市用户，有高频本地生活消费需求：
  - 学生穷游型：价格敏感，更关注性价比。
  - 上班族享受型：对服务质量、口碑和环境要求较高。
  - 宝妈带娃型：对安全性、环境、是否适合儿童更加敏感。
  - 无障碍需求型：对交通便利程度、无障碍设施有要求。

### 3.2 典型使用场景

示例场景包括但不限于：

1. 「帮我找附近 3 公里内评价不错、价格适中的理发店，今晚 8 点前能去的。」
2. 「现在带小孩在商场，推荐附近适合亲子吃饭的餐厅。」
3. 「帮我找一家今晚还开门的洗车店，离我 10 公里以内。」

## 四、功能需求

以下功能需求基于《项目规划.md》中定义的用户故事（US-01 ~ US-22）进行整理和抽象。

### 4.1 搜索与推荐核心流程（Epic 1 / Epic 2 / Epic 4）

#### 4.1.0 Request 模型与生命周期（补充自 Intent Broadcast 版本）

**目标**  
将用户每一次自然语言输入抽象为一个结构化的「Request」对象，作为后续检索、排序以及（可选的）Provider 报价的统一锚点。

**需求要点**

- Request 是一等数据模型，核心字段包括（示例）：
  - `id`：唯一标识；
  - `user_id`：发起请求的用户（如有登录体系，可选）；
  - `service_category`：服务类别（理发/洗车/美甲/餐饮等）；
  - `time_window`：期望服务时间窗口（起止时间或“尽快”）；
  - `location`：位置（当前 GPS / 指定地址 + 半径）；
  - `constraints`：预算、特殊需求等约束条件；
  - `status`：`created` / `in_progress` / `completed` / `expired` 等。
- 自然语言意图识别（4.1.1）除输出意图 JSON 外，还需：
  - 创建一条持久化的 Request 记录；
  - 将 Request 作为主键，用于关联后续候选地点、报价（Offer）等实体。
- Request 生命周期（最简版）：
  - 创建：用户发起查询时生成；
  - 处理中：Agent 流程进行中，可能产生多次中间结果推送；
  - 完成：结果展示给用户且流程结束；
  - 过期：超过一定时间窗口后自动过期，不再接收新的报价/更新。

**验收标准**

- 每次搜索均在后端生成一条 Request 记录，并可通过 ID 关联到对应结果；
- Request 状态随流程推进发生合理变更（至少有 created → completed / expired）。

#### 4.1.1 自然语言意图识别（US-01）

**目标**  
用户可以使用中英或中英混合自然语言输入需求，系统能够识别意图并结构化输出。

**需求要点**

- 支持识别以下信息（不限于）：
  - 服务类型：理发、洗车、美甲、餐饮等；
  - 地理范围：附近、多远、某区域等；
  - 时间：现在、今晚、明天上午等；
  - 预算：便宜一点、预算 100 元内等；
  - 特殊需求：带小孩、适合聚餐、有停车位、无障碍等。
- 输出结构化意图 JSON，示意：

```json
{
  "intent_type": "search",
  "service_category": "haircut",
  "location": {
    "type": "current_gps",
    "radius_km": 3
  },
  "time_window": {
    "type": "today",
    "from": "now",
    "to": "20:00"
  },
  "budget_level": "medium",
  "special_requirements": ["kid_friendly"]
}
```

**验收标准**

- 能正确区分搜索/推荐/详情等意图类型；
- 支持中英文及混合输入的常见表达；
- 意图解析结果下游 Agent 可直接消费。

#### 4.1.2 Top 10 推荐排行榜（US-02）

**目标**  
为每次检索返回最多 10 个候选服务商，按综合推荐分排序。

**需求要点**

- 每条结果至少包含：
  - 名称；
  - 综合评分（模型或规则计算得分）；
  - 距离（与用户当前位置或指定位置的距离）；
  - 价格区间（来自 Places API 数据）；
  - 推荐理由摘要（1–3 条短句）。
- 列表默认按「综合推荐分」降序排序。

**验收标准**

- 结果字段齐全；
- 对于典型查询，前几个结果显著相关；
- 端到端响应时间 < 5s（含 LLM 调用）。

#### 4.1.3 地点详情聚合展示（US-03 / US-04）

**目标**  
为单个地点提供更丰富、可解释的详情视图，便于用户快速判断。

**需求要点**

- 评论聚合：
  - 获取最近约 50 条 Google Reviews；
  - 按星级分组（如 1–2 星、4–5 星）；
  - 提取高频关键词供摘要 Agent 使用。
- 摘要与分星理由：
  - 为好评/差评分别生成 3–5 条核心点评要点；
  - 提供 5 星/1 星等分星理由的聚合摘要。
- 其它信息：
  - 价格区间；
  - 评分分布；
  - 当前营业状态（关联实时验证）。

**验收标准**

- 摘要能准确反映原始评论的主流观点；
- 详情页与列表页信息联动一致；
- 用户能在 1–2 屏内获取关键信息。

#### 4.1.4 实时营业时间与可达性验证（US-07 / US-17 / US-08）

**目标**  
在推荐前自动核查地点是否在可达时间窗口内营业，并给出是否可达的明确提示。

**需求要点**

- 使用 Google Places Details API 获取实时营业状态；
- 使用 Distance Matrix API 计算预计出行时间；
- 根据当前时间 + 出行时间，判断：
  - 营业中；
  - 即将关门（例如 30 分钟内关门且预计行程时间接近上限）；
  - 已关门/不可达。
- 将可达性作为排序权重的一部分：
  - 已关门或严重不可达的地点显著降权或过滤；
  - 即将关门给出明显提示文案。

**验收标准**

- 典型场景中，推荐列表不会出现明显“已关门”但仍排名很靠前的结果；
- 列表条目有可视化的状态标记（营业中/即将关门/已关门）。

#### 4.1.5 Multi-Agent Orchestrator（US-13 ~ US-17）

**目标**  
通过主 Agent（Orchestrator）使用 LangGraph 管理多 Agent 的任务规划与调度。

**需求要点**

- Agent 列表（与架构设计一致）：
  - 主 Agent：意图识别、任务规划、调度与结果聚合；
  - 地点搜索 Agent：封装 Google Places Text Search；
  - 实时验证 Agent：营业状态 + 可达性计算；
  - 评论聚合 Agent：按星级分组评论并抽取关键词；
  - 摘要评分 Agent：生成摘要、分星理由、综合推荐分；
  - 个性化 Agent：结合用户画像与权重进行最终排序。
- 调用流程（简要）：
  - 主 Agent 接收结构化意图；
  - 并行调用：地点搜索 + 实时验证（针对候选集）；
  - 对候选集批量并行评论聚合与摘要评分；
  - 个性化 Agent 对结果重排；
  - 接口层统一对前端输出。
- 降级策略：
  - 某个子 Agent 失败时，只影响部分字段，不阻塞整体返回；
  - 对失败原因进行日志记录。

**验收标准**

- 使用 LangGraph 实现有状态 DAG；
- 至少支持上述节点间的基本并发；
- 能在日志中追踪每个 Agent 的状态；
- 对单个子 Agent 失败有可见的降级行为，而不是整体失败。

### 4.2 前端展示与交互（Epic 1 / Epic 5）

#### 4.2.1 地图可视化与联动（US-05）

**需求要点**

- 使用 Google Maps JS SDK / Mapbox 渲染地图；
- 地图标注与排行榜列表联动：
  - 悬停列表项，高亮对应地图标注；
  - 点击地图标注，高亮对应列表项并滚动到可见区域；
- 支持移动端触控操作（缩放、拖拽等）。

**验收标准**

- 列表与地图联动无明显延迟；
- 移动端手势操作流畅。

#### 4.2.2 服务商详情页（US-06）

**需求要点**

- 从排行榜列表或地图标注点击进入详情页；
- 显示内容：
  - 名称、地址、电话（若有）；
  - 综合评分与评分分布；
  - 评论摘要、分星理由；
  - 价格区间、当前营业状态；
  - 推荐理由列表；
- 动作入口：
  - 跳转官网（外链）；
  - 跳转到 Google Maps（导航）；
  - 查看更多评论（分页加载）。

**验收标准**

- 外链跳转和地图导航均可正常工作；
- 评论列表分页加载无明显卡顿。

#### 4.2.3 扫码访问与 PWA（US-20）

**需求要点**

- 将前端应用打包为 PWA，提供：
  - 移动端友好布局；
  - 可添加到主屏幕的能力（如时间允许）。
- 提供二维码，指向线上部署 URL。

**验收标准**

- 在常见移动设备上，首屏加载时间 < 3 秒（良好网络下）；
- 二维码扫描后可直接访问应用首页。

#### 4.2.4 流式推荐结果输出（US-21）

**需求要点**

- 后端使用 SSE 或 WebSocket 流式输出：
  - 先返回基础地点列表（名称、基础评分、距离等）；
  - 后续逐步补充评论摘要、个性化推荐理由等字段。
- 前端：
  - 逐步渲染列表，采用骨架屏/占位符；
  - 显示加载进度或状态提示。

**验收标准**

- 用户在 2 秒内能看到初步推荐结果；
- 追加的摘要和理由加载时无明显闪烁或布局抖动。

### 4.3 个性化与用户画像（Epic 3）

#### 4.3.1 冷启动偏好问卷（US-09）

**需求要点**

- 首次使用时弹出 ≤ 5 个问题的偏好问卷，包含：
  - 预算水平：低 / 中 / 高；
  - 距离敏感度：更近优先 / 可接受适度距离换更好服务；
  - 类型偏好：价格优先 / 评分优先 / 品牌优先；
  - 典型场景偏好：学生穷游 / 宝妈带娃 / 上班族享受等；
  - 是否有无障碍等特殊需求。
- 用户可以选择“跳过”，则加载默认权重配置；
- 问卷结果存储在用户 Profile（如 Supabase）。

**验收标准**

- 问卷流程不影响首屏核心体验（可延后/底部弹出）；
- 完成问卷后，后续推荐结果明显体现其偏好。

#### 4.3.2 偏好权重自定义（US-10）

**需求要点**

- 在设置或侧边栏中提供滑块/选项，用于调节以下权重：
  - 价格敏感度；
  - 距离权重；
  - 评分权重；
  - 品牌/人气权重。
- 每次调整后，可触发重新排序请求，列表实时刷新；
- 偏好配置持久化到用户 Profile。

**验收标准**

- 调整权重后，列表排序变化与预期一致；
- 页面交互延迟在可接受范围内。

#### 4.3.3 用户画像分类与策略（US-11）

**需求要点**

- 至少支持 3 种以上画像策略（例如：学生型、省钱型、享受型、带娃型）；
- 不同画像对应不同的默认排序权重配置；
- 用户可以在设置中手动切换画像类型。

**验收标准**

- 画像切换后，对典型查询结果影响明显；
- 配置变更被记录并在后续调用中生效。

#### 4.3.4 推荐理由透明化（US-12）

**需求要点**

- 每条推荐需附带 1–3 条简洁的推荐理由；
- 推荐理由中需要体现：
  - 与用户当前偏好/画像的匹配点；
  - 与实时状态的关系（如“当前营业中”“距您仅 1.2km”等）。

**验收标准**

- 推荐理由自然易读，避免“AI 腔”；
- 用户能够通过阅读推荐理由理解排序依据。

## 五、非功能性需求

### 5.1 性能

- 常规查询场景端到端响应时间 < 5 秒；
- 使用流式输出时，首批结果（基础列表）尽量在 2 秒内返回；
- 多 Agent 并发执行应控制在合理并发度，避免 API 限流。

### 5.2 可用性与体验

- 支持中英文混合输入；
- 移动端优先设计，关键操作单手即可完成；
- 错误提示清晰友好，给出“重试”或“更换条件”等建议。

### 5.3 可靠性

- Google Places API 调用失败时：
  - 提示用户稍后重试或使用部分缓存结果；
  - 记录失败日志，避免无限重试。
- LLM 调用超时或失败时：
  - 仍返回基础地点列表信息；
  - 缺失的摘要/理由以占位提示替代。

### 5.4 安全与合规

- 仅请求必要权限（例如位置权限），并在 UI 中清晰说明用途；
- 不存储敏感隐私信息；
- 日志中避免记录用户精确坐标或敏感文本。

## 六、技术架构概要（与需求相关）

> 技术细节以架构/技术设计文档为准，此处只列与 PRD 关联的关键选型。

- 前端：
  - 框架：Next.js 14 + Tailwind CSS；
  - 地图：Google Maps JS SDK / Mapbox GL；
  - 形态：H5 + PWA，部署至 Vercel。
- 后端：
  - 接口层：FastAPI（Python），提供 REST + SSE 接口；
  - Agent 框架：LangGraph（推荐）+ LLM（GPT-4o-mini / Claude Haiku）；
  - Orchestrator 与各子 Agent 以内部服务形式提供。
- 数据源与存储：
  - Google Places API（Text Search / Details / Reviews / Distance Matrix）；
  - Supabase（用户画像、偏好、基础日志）。

### 6.1 Provider 端与 Offer 模型（补充自 Provider Bid/Offer 版本）

> 本小节为在 MVP 基础上拓展的“双边撮合”能力，供团队权衡范围使用。若当期不实现，可视为 V1.5/V2 特性。

**Provider 角色**

- 引入 `provider` 角色（商家/服务方），最小能力包括：
  - 接收一定范围内的 Request 通知（例如：距离、服务类别匹配的请求）；
  - 针对某个 Request 提交报价（Offer）。

**Offer 数据模型（简化示例）**

- 关键字段：
  - `id`：唯一标识；
  - `request_id`：关联的 Request；
  - `provider_id`：报价方；
  - `price`：报价价格；
  - `eta`：预计可服务时间/到达时间；
  - `slot`：可预约时间段（如 18:00–19:00）；
  - `status`：`pending` / `accepted` / `rejected` / `expired`。

**最小闭环能力**

- Provider 端（可为简化 Web 控制台或 Mock 接口）：
  - 能看到附近匹配的 Request 列表；
  - 能对单个 Request 提交 Offer。
- 用户端：
  - 能看到某个 Request 下的 Offer 列表，按综合 score 排序（可复用 4.1.2 排序逻辑）。

**实时更新（与 6.2 呼应）**

- 新的 Offer 通过实时通道（SSE/WebSocket）推送到用户前端，形成“竞价”体验。

### 6.2 实时报价与结果更新（补充自 Realtime 更新版本）

在 4.2.4 已定义“流式推荐结果输出”（同一请求的计算过程逐步完成），本节补充 **新的报价/结果的实时加入** 能力：

- 当新的 Offer 被创建或已有 Offer 状态发生变化时：
  - 通过 SSE/WebSocket 通知关联 Request 的前端会话；
  - 前端在当前列表中插入/更新对应 Offer 卡片，并保持排序正确。
- 若当前阶段尚未实现 Provider 真实端，也可用模拟服务/定时任务生成新的 Offer，用于演示实时刷新体验。

**验收标准**

- 在同一 Request 页面停留期间，可看到新的 Offer 实时出现在列表中，无需手动刷新；
- 排序与解释（推荐理由）仍然保持正确且稳定。

## 八、风险、假设与数据/可观测性补充

### 8.0 Demo 种子数据（补充自 Data Model & Seed 版本）

- 为了保证在 API 受限或网络不稳定时仍能完成 Demo，需准备一份 **固定种子数据集**，覆盖至少一个城市/区域（例如：苏黎世某个街区，或目标城市的一个商圈）。
- 建议数据规模：**15–30 家店**，字段包括但不限于：
  - 名称、类别；
  - 经纬度；
  - 营业时间；
  - 评分、价格区间；
  - 简要标签/关键词（可用于 Mock 评论摘要）。
- 使用方式：
  - 作为 Google Places API 的补充或 fallback；
  - 在演示场景中可强制走种子数据，以确保可重复性。

### 8.1 主要风险

- Google Places API 的调用配额与延迟可能影响体验；
- LLM 在多地点并行摘要时的成本与时延较高；
- Multi-Agent 调度逻辑复杂，调试与观测难度较大。

### 8.2 关键假设

- 当前版本主要用于单城市范围内的 Demo 和内测；
- 用户规模有限，无大规模并发压力；
- 团队成员具备基本的 Next.js / FastAPI / LangChain / LangGraph 开发经验。

### 8.3 Agent Trace 与可观测性（补充自 Agent Trace 版本）

- 为方便评审和调试，建议在前端提供一个隐藏或只读的 **Debug Panel**：
  - 展示主 Agent 的任务规划（DAG 结构或 JSON 计划）；
  - 展示各子 Agent 的关键输入/输出摘要（可折叠查看原始 JSON）。
- Orchestrator 侧需：
  - 为每次 Request 生成可追踪的 trace ID；
  - 按节点记录执行状态、耗时和错误信息。
- Debug Panel 可以：
  - 在特定参数（如 `?debug=true`）或特定用户下开启；
  - 作为演示时展示“智能过程”的重要工具。

### 8.4 安全与隐私最小化（补充自 Security & Privacy 版本）

- 在 5.4 的基础上，增加 **用户可感知的隐私说明与数据最小化实践**：
  - 提供一个“隐私与数据使用”页面或弹窗，简要说明：
    - 会请求哪些权限（定位等）；
    - 会存哪些数据（偏好配置、画像标签、基础日志）；
    - 不会存哪些数据（原始对话文本、精确历史轨迹等，若适用）。
  - 在第一次请求关键权限时，给出用途说明（例如“用于推荐离你最近、目前营业的服务商”）。
- 后端需确保：
  - 日志中不记录用户精确坐标和可识别身份信息；
  - 支持按用户 ID 查阅/删除其偏好与画像数据（如未来有账号体系）。
