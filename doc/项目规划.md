## 一、项目概述

本系统是基于 **6-Agent 架构**的智能出行顾问平台。用户通过自然语言输入需求，系统由 6 大 Agent 模块协作完成：Input Agent（意图理解）、Crawling Agent（Apify 店铺搜索 + SBB 公共交通 ETA）、Evaluation Agent（评分计算）、Review Agent（Apify 评论抓取 + LLM 摘要）、Orchestrator Agent（结果聚合与可解释推荐）、Output Agent（排名列表 + 一句话推荐），最终返回可解释的个性化推荐排行榜，并在地图上可视化展示。

**核心差异化定位：**

- 相比直接问 Gemini/ChatGPT：提供地图 + 评论 + 实时数据一体化垂直体验
- 相比 Google Maps：提供个性化排序 + LLM 优劣势摘要 + 推荐理由解释 + SBB 公共交通路线
- 技术亮点：6-Agent DAG 协作架构（含并行分支），集成 Apify Google Maps scraper + SBB API，展示 LangGraph 有状态工作流能力

## 三、Multi-Agent 架构设计

### 3.1 Agent 一览（6-Agent 架构）

| Agent | 职责 | 调用工具/API | 输出 |
|-------|------|-------------|------|
| 🧠 **Input Agent** | 解析用户自然语言输入，输出结构化意图 JSON | LLM (GPT-4o/Claude) | 结构化意图（服务类型、位置、时间窗口、预算、特殊需求） |
| 🔍 **Crawling Agent — Sub-1（Store Crawler）** | 根据用户位置和意图搜索候选店铺，过滤营业时间不匹配的店铺 | **Apify Google Maps scraper** | 候选店铺列表（含营业时间） |
| 🚃 **Crawling Agent — Sub-2（Transit Calculator）** | 查询用户到每个候选店铺的公共交通路线 | **SBB API** | 每店铺的 TransitInfo（duration、transport_types、departure_time、connections） |
| ⭐ **Evaluation Agent** | 根据用户偏好权重计算每个店铺的综合推荐分 | 评分算法 + 用户偏好权重 | 带分数的候选列表 |
| 📝 **Review Agent** | 批量抓取所有候选店铺评论，LLM 生成优势/劣势摘要 | **Apify Google Maps review scraper** + LLM | 每店铺的 advantages / disadvantages 摘要 |
| 🎯 **Orchestrator Agent** | 收集 Evaluation 和 Review 输出，整合为可解释的最终推荐列表 | LLM + 聚合逻辑 | 精炼后的 Top N 推荐 + 推荐理由 |
| 📊 **Output Agent — Sub-1（Ranking Formatter）** | 输出最终排名列表 | 格式化逻辑 | 排名列表（名称、评分、距离、交通、价格、推荐理由） |
| 💬 **Output Agent — Sub-2（Recommendation Writer）** | 为每个店铺生成一句话推荐文案 | LLM | one_sentence_recommendation |

### 3.2 调用流程（高层 DAG）

```
用户输入
  → ① Input Agent（自然语言解析 → 结构化意图）
  → 并行分支：
      ├─ ② Crawling Agent Sub-1（Apify 店铺搜索 + 营业时间过滤 → 候选列表）
      │    └─ ③ Crawling Agent Sub-2（SBB 公共交通 ETA → TransitInfo）
      └─ ④ Review Agent（Apify 评论抓取 + LLM 优劣势摘要 → advantages/disadvantages）
  → 汇合
      ⑤ Evaluation Agent（基于偏好权重计算每店铺分数 → 带分数列表）
      ⑥ Orchestrator Agent（聚合 ⑤+④ 结果，精炼可解释推荐列表）
      ⑦ Output Agent Sub-1（格式化排名列表）
      ⑧ Output Agent Sub-2（LLM 生成 one_sentence_recommendation）
  → 接口层聚合
  → 流式 SSE 推送前端（7 阶段事件：intent_parsed → stores_crawled → transit_computed
      → reviews_fetched → scores_computed → recommendations_ready → completed）
```

### 3.3 技术框架：LangGraph

推荐使用 **LangGraph** 实现 6-Agent DAG 调度，核心优势：

- **节点（Node）** = 每个 Agent / Sub-Agent，**边（Edge）** = 依赖关系
- 支持并发节点执行（Crawling 分支与 Review 分支可并行）
- Crawling Sub-2 依赖 Sub-1 完成后才执行（串行子图）
- 内置状态管理，便于调试和可视化 Agent 执行路径（参见 `/traces/{trace_id}` 接口）
- 与 LangChain 工具生态无缝集成

| **Sprint 4**（第7天） | UI打磨，移动端适配，演示脚本 | 性能调优，演示场景准备 | Edge case处理，演示数据准备 | 部署上线，二维码生成，监控 | 线上可访问，演示Ready |

**⚠️ 风险提示：**

- Apify scraper 有调用配额限制，建议监控用量并设置缓存（Redis/内存缓存）
- SBB API 在高峰时段响应可能较慢，需设置合理超时和降级策略
- LLM 调用延迟可能影响体验，Review Agent 和 Output Agent 摘要任务用小模型（Claude Haiku / GPT-4o-mini）并做流式输出
- 6-Agent DAG 调度复杂度高，Sprint 1 优先跑通单条调用链，Sprint 2 再做 Crawling/Review 并发优化

## 六、技术栈汇总

| 层级 | 推荐选型 | 备注 |
|------|---------|------|
| 前端框架 | Next.js 14 + Tailwind CSS | PWA支持，部署至Vercel |
| 地图 | Google Maps JS SDK / Mapbox GL | 用于前端地图可视化展示 |
| Agent框架 | **LangGraph**（推荐）/ LangChain AgentExecutor | LangGraph 支持有状态 DAG，6-Agent 并行分支 |
| LLM | GPT-4o-mini（速度）/ Claude Haiku（性价比） | 摘要任务（Review Agent、Output Agent）用小模型降低成本 |
| 店铺搜索 | **Apify Google Maps scraper** | 替代 Google Places Text Search，获取店铺列表 + 营业时间 |
| 评论抓取 | **Apify Google Maps review scraper** | 替代 Google Places Reviews API，批量抓取评论文本 |
| 公共交通 | **SBB API** | 替代 Distance Matrix API，返回 duration、transport_types、departure_time、connections |
| 后端框架 | FastAPI (Python) | 原生异步支持，SSE友好 |
| 用户存储 | Supabase（推荐）/ Firebase | 免费额度足够一周项目 |
| 实时推送 | Server-Sent Events (SSE) | 7 阶段流式事件（intent_parsed → completed） |
| 部署 | Vercel（前端）+ Railway/Render（后端） | 均有免费额度 |
