## 一、项目概述

本系统是基于 **Multi-Agent 架构**的智能出行顾问平台。用户通过自然语言输入需求，系统由主 Agent（Orchestrator）进行任务规划，协调多个专职子 Agent 并发执行搜索、实时验证、评论聚合、摘要评分、个性化排序等任务，最终返回可解释的个性化推荐排行榜，并在地图上可视化展示。

**核心差异化定位：**

- 相比直接问 Gemini/ChatGPT：提供地图 + 评论 + 实时数据一体化垂直体验
- 相比 Google Maps：提供个性化排序 + LLM 摘要 + 推荐理由解释
- 技术亮点：多 Agent 协作架构，展示 LangGraph 有状态工作流能力

## 三、Multi-Agent 架构设计

### 3.1 Agent 一览

| Agent | 职责 | 调用工具/API | 输出 |
|-------|------|-------------|------|
| 🧠 **主Agent（Orchestrator）** | 解析用户意图，制定任务DAG，调度子Agent，聚合最终结果 | LLM (GPT-4o/Claude) · LangGraph | 任务计划JSON / 最终推荐结果 |
| 🔍 **地点搜索 Agent** | 根据意图+位置搜索候选地点 | Google Places Text Search API · Geolocation API | 候选地点列表（标准化） |
| ⏰ **实时验证 Agent** | 核查营业时间、计算可达时间、标注时效状态 | Google Places Details API · Distance Matrix API | 时效标签 + 备选提示 |
| 📝 **评论聚合 Agent** | 获取评论，按星级分组，提取关键词 | Google Places Reviews API | 分组评论 JSON |
| ✨ **摘要评分 Agent** | LLM生成好/差评摘要、分星理由、综合推荐分 | LLM + 个人偏好权重 | 摘要JSON + 推荐分 + 推荐理由 |
| 🎯 **个性化 Agent** | 根据用户画像调整权重，重新排序结果 | 用户Profile存储 | 最终排序后的 Top 10 |

### 3.2 调用流程（高层）

```
用户输入
  → 主Agent（意图识别 + 任务规划）
  → 并发调度子Agent：
      ① 地点搜索Agent      → 候选地点列表
      ② 实时验证Agent      → 营业状态 + 可达时间（与①并发）
  → 合并结果，过滤不可达地点
      ③ 评论聚合Agent      → 分组评论（对每个候选地点）
      ④ 摘要评分Agent      → 好/差评摘要 + 推荐分（基于③）
      ⑤ 个性化Agent        → 融合用户偏好权重，最终排序
  → 接口层聚合
  → 流式推送前端
```

### 3.3 技术框架：LangGraph

推荐使用 **LangGraph** 实现主Agent的有状态DAG调度，核心优势：

- **节点（Node）** = 每个子Agent，**边（Edge）** = 依赖关系
- 支持并发节点执行（③④可在②完成后批量并发）
- 内置状态管理，便于调试和可视化Agent执行路径
- 与 LangChain 工具生态无缝集成

**⚠️ 风险提示：**

- Google Places API 有调用配额限制，建议提前申请并设置缓存（Redis/内存缓存）
- LLM调用延迟可能影响体验，摘要任务用小模型（Claude Haiku / GPT-4o-mini）并做流式输出
- Agent调度复杂度高，Sprint 1 优先跑通单条调用链，Sprint 2 再做并发优化

#### US-23 · 用户请求建模（Request 对象）

> 作为系统，我希望将每次自然语言输入抽象为一个结构化的 Request 对象，以便后续检索、排序和报价都能围绕统一实体进行。

| 字段 | 内容 |
|------|------|
| **责任人** | 🟢 后端D（接口层） |
| **优先级** | 中 |
| **MVP** | ✅ |
| **Sprint** | Sprint 1 |

**验收标准：**

- 每次搜索会在后端创建一条 Request 记录，包含服务类别、时间窗口、位置、约束等核心字段；
- Request 状态可从创建 → 完成 / 过期，便于日志和后续分析；
- 所有关联结果（候选地点、报价等）均可通过 Request ID 关联。

---

#### US-24 · 服务商接单与报价（Provider Bid/Offer）

> 作为服务商，我希望收到附近用户的 Request 并提交报价（价格/可服务时间），以便填补空闲时段。

| 字段 | 内容 |
|------|------|
| **责任人** | 🟢 后端D（接口层） + 🔵 前端（简单 Provider 控制台，若实现） |
| **优先级** | 低（可视为 V1.5/V2） |
| **MVP** | ⚠️ 可选 |
| **Sprint** | V2 |

**验收标准：**

- Provider 端可以看到一定范围内匹配的 Request 列表；
- Provider 可以针对单个 Request 提交 Offer（包含价格、ETA/时间段）；
- 用户侧可以看到对应 Request 下的 Offer 列表，按综合 score 排序。

---

#### US-25 · 实时报价更新（Realtime Offers）

> 作为用户，当我停留在某个请求页面时，希望看到新的报价实时出现，而不是手动刷新页面。

| 字段 | 内容 |
|------|------|
| **责任人** | 🟢 后端D（SSE/WebSocket） + 🔵 前端 |
| **优先级** | 低（依赖 US-24） |
| **MVP** | ⚠️ 可选 |
| **Sprint** | V2 |

**验收标准：**

- 新的 Offer 创建或状态变化时，会实时推送到当前 Request 的前端会话；
- 列表能插入/更新对应报价卡片并保持排序正确；
- Demo 场景中可通过真实 Provider 或模拟任务看到“竞价”效果。

### Epic 7：数据与可观测性

> 作为团队，我希望为一个固定区域准备 15–30 家服务商的种子数据，用于在 API 受限或网络不稳定时也能完整演示端到端流程。

| 字段 | 内容 |
|------|------|
| **责任人** | 🟡 后端C（推荐与评分） + 全团队 |
| **优先级** | 中 |
| **MVP** | ✅ |
| **Sprint** | Sprint 2 |

**验收标准：**

- 至少包含 15–30 家店，字段包括：名称、类别、经纬度、营业时间、评分、价格区间等；
- 后端可在配置中切换“真实 API / 种子数据 / 混合模式”；
- 演示脚本中至少有 1–2 个场景完全基于种子数据可重复复现。

#### US-27 · Agent Trace 调试面板

> 作为评审/开发者，我希望在演示时看到主 Agent 的任务规划和子 Agent 的中间输出，以便理解系统智能行为和排查问题。

| 字段 | 内容 |
|------|------|
| **责任人** | 🔴 后端B（Agent Orchestrator） + 🔵 前端 |
| **优先级** | 中 |
| **MVP** | ⚠️ 简化版 |
| **Sprint** | Sprint 3 |

**验收标准：**

- 为每次请求生成 trace ID，后端记录各 Agent 的输入/输出摘要和耗时；
- 前端提供隐藏或参数触发的 Debug Panel（例如 `?debug=true`），展示：
  - 主 Agent 规划的 DAG 或步骤列表；
  - 各子 Agent 的简要 JSON 输出（可折叠查看）；
- Demo 时可通过该面板讲解“系统是如何思考和决策的”。

### Epic 8：安全与隐私

#### US-28 · 数据最小化与隐私说明

> 作为用户，我希望系统尽量少收集个人数据，并清楚告知具体会存哪些信息，以便放心使用。

| 字段 | 内容 |
|------|------|
| **责任人** | 🔵 前端 + 🟢 后端D |
| **优先级** | 中 |
| **MVP** | ✅ |
| **Sprint** | Sprint 3 |

**验收标准：**

- 提供“隐私与数据使用”简要页面或弹窗，说明：
  - 会请求哪些权限（如定位）；
  - 会存哪些数据（偏好配置、画像标签、基础日志）；
  - 不会存哪些数据（敏感个人身份信息、精确历史轨迹等）。
- 首次请求关键权限时给出用途说明（如“用于推荐离你最近、当前营业的商家”）；
- 后端日志中不记录可识别个人身份的敏感信息。

## 六、技术栈汇总

|------|---------|------|
